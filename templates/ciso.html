Discovery · How it works

{{ seed_domain_count }} seed domain{% if seed_domain_count != 1 %}s{% endif %}. Passive signals. No credentials.

Everything in this report was derived passively from {% for d in seed_domains_preview %}{{ d }}{% if not loop.last %} and {% endif %}{% endfor %}{% if seed_domains_overflow %} and {{ seed_domains_overflow }} more{% endif %} — no agents, no authenticated requests, no prior asset inventory. Veracode EASM chains four publicly-observable data sources to build a complete picture of the external estate.

01 — DNS
Subdomain enumeration
Certificate transparency logs, passive DNS, and zone records surface subdomains, CNAME chains, and NS delegations — the {{ unique_fqdns }} FQDNs and {{ total_cnames }} CNAME records in this report.
02 — TLS / PKI
Certificate mapping
Every certificate on a live endpoint is inspected for SANs, issuer, and hostname match — surfacing mismatches and mapping which certificate authority is in use per asset.
03 — IP & ASN
Network attribution
IP addresses are attributed to ASNs and mapped to suppliers — cloud providers, CDNs, and hosting vendors. {{ bare_ip_count }} bare IPs were found without associated hostname records.
04 — HTTP headers & software
Technology fingerprinting
A passive HTTP probe of each live endpoint reads response headers and server banners — identifying missing security headers across {{ total_apps }} applications.

How supplier connections are inferred: When a CNAME resolves to a third-party edge or CDN, an SPF or MX record names an email provider, or a response header carries a cloud or platform signature — EASM records a supplier connection vector. The {{ suppliers | length }} suppliers in this report were identified entirely from these passive signals, not from contract or procurement data.
